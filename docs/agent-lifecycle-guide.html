Cortex Agent Lifecycle with dbt

A dbt project template that manages the full Cortex Agent development lifecycle as code — semantic views, agent configurations, evaluations, and scheduling — all running natively inside Snowflake.

View the full repository on GitHub

Starting point

This flow begins after business questions and the data models that can answer them have already been identified. The template operationalizes that work into a repeatable, version-controlled pipeline.

Data Foundation

Steps 1 and 2 establish the reliable source and staging layer that the semantic view will be built on. Every downstream object — the semantic view, the agent, the evaluations — inherits the quality of this foundation.

Step 1 & 2 Walkthrough

1 Define Sources — models/sources.yml
View sources.yml on GitHub

Edit models/sources.yml to register your raw tables. dbt uses this file to resolve {{ source() }} calls throughout the project with fully-qualified names.

What to change: Replace raw_database, raw_schema, and the table names with your actual source values.

version: 2

sources:
  - name: raw                          # logical name used in {{ source('raw', ...) }}
    database: raw_database             # TODO: your source database
    schema: raw_schema                 # TODO: your source schema
    tables:
      - name: orders                   # TODO: add your tables
        description: Raw order records
      - name: products
        description: Product catalog

Why sources matter

Declaring sources enables dbt lineage — Snowflake can trace data from your Cortex Agent's semantic view all the way back to the raw tables it depends on. It also lets dbt source freshness alert you when upstream data stops updating.

2 Build Staging Models — models/staging/*.sql
View the staging directory on GitHub

Create one .sql file per source table in models/staging/. Staging models do three things: enforce data types, clean nulls/whitespace, and rename columns to consistent business names. The semantic view references {{ ref('stg_orders') }} — never the raw source directly.

Naming convention: stg_{source_name}_{table_name}.sql

-- models/staging/stg_orders.sql

SELECT
    order_id::VARCHAR          AS order_id,
    customer_id::VARCHAR       AS customer_id,
    order_date::DATE           AS order_date,
    TRIM(status)               AS status,
    order_total::FLOAT         AS order_total,
    COALESCE(region, 'Unknown') AS region
FROM {{ source('raw', 'orders') }}
WHERE order_id IS NOT NULL
# Build only staging models
dbt build --select staging.*

# Build a single staging model
dbt build --select stg_orders

Keep staging models thin

Staging models should do no aggregation or business logic — just type casts, null handling, and renaming. Business logic lives in the semantic view's METRICS and DIMENSIONS blocks, where Cortex Analyst can reason about it.

Semantic View

Step 3 translates your staging models into a business-readable semantic layer. The semantic view is what Cortex Analyst reads to convert natural language questions into SQL — every fact, dimension, metric, and verified query you define directly improves answer accuracy.

How materialized='semantic_view' works

The dbt_semantic_view package intercepts this materialization type and passes your SQL body verbatim to Snowflake's CREATE OR REPLACE SEMANTIC VIEW statement. You get full SQL API coverage for every Snowflake feature release without waiting for a package upgrade.

Step 3 Walkthrough

1 dbt config block — top of every semantic view
View sv_example.sql on GitHub

Every file in models/semantic_views/ must start with this config. The name you give the model becomes the semantic view's object name in Snowflake.

{{ config(materialized='semantic_view') }}

File path: models/semantic_views/sv_orders.sql → creates {database}.{schema}.SV_ORDERS

2 TABLES block — register logical tables with primary keys

The TABLES block maps logical table aliases to your dbt models. Use {{ ref() }} for staging models and {{ source() }} for raw tables. Primary keys are required — they tell Cortex Analyst how to deduplicate rows in aggregations.

TABLES (
  orders AS {{ ref('stg_orders') }}
    PRIMARY KEY (order_id)
    COMMENT = 'Customer orders with fulfillment status and revenue',

  products AS {{ ref('stg_products') }}
    PRIMARY KEY (product_id)
    COMMENT = 'Product catalog with categories and pricing'
)
3 RELATIONSHIPS block — define foreign key joins

Relationships tell Cortex Analyst how to join tables when a question spans multiple entities. Without this, the model can only answer single-table questions.

RELATIONSHIPS (
  orders_to_products AS
    orders (product_id) REFERENCES products
)
4 FACTS block — row-level expressions

A fact is a value evaluated at the individual row level — not aggregated. Metrics are built on top of facts: the fact order_value holds the per-row amount, and a metric like SUM(order_value) aggregates it. The FACTS clause must appear before DIMENSIONS and METRICS in the statement.

FACTS (
  orders.order_value AS order_total
    COMMENT = 'Per-order revenue amount, before aggregation',

  orders.is_fulfilled AS IFF(status = 'Fulfilled', 1, 0)
    COMMENT = 'Row-level 1/0 flag used to compute fulfillment rate'
)

Facts vs. metrics

Think of a fact as the raw per-row number and a metric as the aggregation over it. Defining the row-level value once as a fact lets multiple metrics (sum, average, ratio) reuse it consistently. Facts can also be marked PRIVATE when they exist only to feed a metric and shouldn't be queryable on their own.

5 DIMENSIONS block — filterable and groupable attributes

Dimensions are columns users will filter by or group on ("by region", "for Q1", "where status is Fulfilled"). SYNONYMS expand the vocabulary Cortex Analyst recognizes for each dimension — add the natural language terms your users actually say.

DIMENSIONS (
  orders.order_date AS order_date
    COMMENT = 'Date the order was placed'
    WITH SYNONYMS = ('purchase date', 'transaction date', 'order day'),

  orders.status AS order_status
    COMMENT = 'Fulfillment status of the order'
    WITH SYNONYMS = ('fulfillment status', 'order state'),

  orders.region AS region
    COMMENT = 'Geographic region of the order'
    WITH SYNONYMS = ('territory', 'area', 'geography'),

  products.product_name AS product_name
    COMMENT = 'Display name of the product'
    WITH SYNONYMS = ('item name', 'product', 'sku name')
)
6 METRICS block — aggregate measures

Metrics are the numbers users ask about ("total revenue", "order count", "average order value"). They aggregate over the facts and columns defined above — note how total_revenue sums the order_value fact. Define them here rather than in staging so Cortex Analyst understands them semantically, not just syntactically.

METRICS (
  orders.total_revenue AS SUM(order_value)
    COMMENT = 'Total revenue from orders'
    WITH SYNONYMS = ('revenue', 'sales', 'total sales', 'gross revenue'),

  orders.order_count AS COUNT(order_id)
    COMMENT = 'Number of orders placed'
    WITH SYNONYMS = ('number of orders', 'orders', 'order volume'),

  orders.fulfillment_rate AS AVG(is_fulfilled)
    COMMENT = 'Share of orders that reached Fulfilled status'
    WITH SYNONYMS = ('fulfillment percentage', 'fulfilled rate')
)
7 AI_SQL_GENERATION block — business rules for generated SQL

This block injects business-specific guidance into every SQL statement Cortex Analyst generates. Use it to encode conventions that aren't expressible in the schema — default date handling, rounding rules, or required filters — so answers stay consistent without the user having to specify them.

AI_SQL_GENERATION
  'Default to the most recent complete month if no date range is specified.
   Always round currency values to 2 decimal places.
   Exclude cancelled orders (status = ''Cancelled'') from all revenue metrics
   unless the user explicitly asks to include them.'

When to use it

Reach for AI_SQL_GENERATION whenever you find yourself correcting the same class of SQL mistake repeatedly — it's the place to codify "house rules" that apply across many questions rather than a single verified query.

8 AI_QUESTION_CATEGORIZATION block — out-of-scope guardrails

This block defines guardrails that classify and reject questions outside the semantic view's domain. Instead of hallucinating an answer for a question the model can't support, the agent declines cleanly — critical for trust when the view is shipped to business users.

AI_QUESTION_CATEGORIZATION
  'Reject questions about employee data, HR topics, or individual
   customer PII. Reject questions that require data not present in
   this semantic view (for example, marketing spend or web traffic).
   For rejected questions, briefly explain what this model can answer
   instead.'

Guardrails build trust

A model that confidently answers questions it has no data for erodes user trust faster than one that politely declines. Defining the out-of-scope boundary explicitly is one of the highest-impact steps before go-live.

9 AI_VERIFIED_QUERIES block — ground-truth SQL pairs

Verified queries are the single highest-leverage addition to a semantic view. Each pair tells Cortex Analyst exactly what SQL to generate for a known question — improving accuracy for similar questions through retrieval-augmented generation. Add the questions you know users will ask most.

AI_VERIFIED_QUERIES (
  top_products AS (
    QUESTION 'What are the top selling products this quarter?'
    VERIFIED_AT 1747267200
    ONBOARDING_QUESTION TRUE
    SQL 'SELECT product_name, total_revenue
         FROM SEMANTIC_VIEW(sv_orders
           DIMENSIONS products.product_name
           METRICS orders.total_revenue)
         ORDER BY total_revenue DESC
         LIMIT 10'
  ),

  revenue_by_region AS (
    QUESTION 'What is total revenue by region?'
    VERIFIED_AT 1747267200
    ONBOARDING_QUESTION TRUE
    SQL 'SELECT region, total_revenue
         FROM SEMANTIC_VIEW(sv_orders
           DIMENSIONS orders.region
           METRICS orders.total_revenue)
         ORDER BY total_revenue DESC'
  )
)

How many verified queries to add?

Start with 5–10 covering the most common question patterns for your domain. Each one acts as a retrieval anchor — questions semantically similar to a verified query will reuse its SQL pattern, dramatically improving accuracy beyond the exact match.

10 Build and deploy the semantic view 
# Build a single semantic view
dbt build --select sv_orders

# Build all semantic views
dbt build --select semantic_views.*

# Verify the object was created in Snowflake
SHOW SEMANTIC VIEWS LIKE 'SV_ORDERS';

After building, the fully-qualified semantic view name (e.g., DEV_DB.CORTEX_AGENTS.SV_ORDERS) is what you'll reference in your agent YAML spec under tool_resources.Analyst.semantic_view.

Agent Configuration

Step 4 takes the semantic view you built and wires it into a Cortex Agent. The agent spec is a YAML file — the create_agent macro reads it and issues CREATE OR REPLACE AGENT on your behalf. Updates are zero-downtime via alter_agent.

Step 4 Walkthrough

1 Write the agent spec — agents/example_agent.yml
View example_agent.yml on GitHub

Create one YAML file per agent in the agents/ directory. The filename (without .yml) is the agent's object name in Snowflake.

models:
  orchestration: auto        # Snowflake picks the best orchestration model

orchestration:
  budget:
    seconds: 30              # Hard cap on time per request
    tokens: 16000            # Hard cap on orchestration tokens per request

instructions:
  response: >
    You are a data analyst assistant for [your use case].
    Provide clear, data-driven answers. Always cite specific
    numbers and dates. Round currency to 2 decimal places.
  orchestration: >
    For quantitative questions about orders, revenue, and
    fulfillment, use the Analyst tool. For questions about
    business definitions or policies, use the Search tool.
    Never guess numeric values — always use a tool.
  sample_questions:
    - question: "What were total sales last month?"
    - question: "Which products had the highest revenue in Q1?"
    - question: "How many orders are pending fulfillment today?"

tools:
  - tool_spec:
      type: "cortex_analyst_text_to_sql"
      name: "Analyst"
      description: >
        Queries structured order, inventory, and product data
        to answer questions about revenue, demand, and
        fulfillment metrics. Do NOT use for definitions or
        policy questions.

tool_resources:
  Analyst:
    semantic_view: "DEV_DB.CORTEX_AGENTS.SV_ORDERS"

Tool descriptions drive routing accuracy

The orchestration model reads tool descriptions to decide which tool to call. Be explicit about what each tool does and what it does not do. Vague descriptions are the primary cause of misrouted queries.

Adding Cortex Search (optional)

To add unstructured document search alongside the Analyst tool, add a second tool spec:

  - tool_spec:
      type: "cortex_search"
      name: "Search"
      description: >
        Searches business documentation for definitions,
        policies, and terminology. Do NOT use for numeric
        queries or data that requires aggregation.

tool_resources:
  Search:
    name: "DEV_DB.CORTEX_AGENTS.MY_SEARCH_SERVICE"
    max_results: "5"
2 Create the agent — macros/create_agent.sql
View create_agent.sql on GitHub

The create_agent macro reads your YAML spec from disk using load_file_contents() and issues CREATE OR REPLACE AGENT ... FROM SPECIFICATION $$...$$. The agent name and fully-qualified location are derived from target.database and target.schema in your profiles.yml.

{% macro create_agent(agent_name, spec_file=none) %}
  {%- set db = target.database -%}
  {%- set schema = target.schema -%}
  {%- if spec_file is none -%}
    {%- set spec_file = 'agents/' ~ agent_name ~ '.yml' -%}
  {%- endif -%}
  {%- set spec_content = load_file_contents(spec_file) -%}

  {% set sql %}
    CREATE OR REPLACE AGENT {{ db }}.{{ schema }}.{{ agent_name }}
      COMMENT = 'Managed by dbt - do not edit manually'
      FROM SPECIFICATION $${{ spec_content }}$$
  {% endset %}

  {% do run_query(sql) %}
{% endmacro %}
# Create or replace the agent
dbt run-operation create_agent --args '{agent_name: example_agent}'

# Verify the agent was created
SHOW AGENTS LIKE 'EXAMPLE_AGENT';
3 Update a live agent — zero-downtime with alter_agent
View alter_agent.sql on GitHub

After the initial creation, use alter_agent instead of create_agent to update the live agent's specification without interrupting active sessions. This issues ALTER AGENT MODIFY LIVE VERSION, which applies changes atomically.

# Update the live agent in-place
dbt run-operation alter_agent --args '{agent_name: example_agent}'

When to use create_agent vs alter_agent

Use create_agent the first time or when you need a full reset. Use alter_agent for incremental updates to instructions, tool descriptions, or budget settings on an already-deployed agent — it avoids downtime and preserves version history.

4 Test the agent interactively

Once deployed, test the agent directly in Snowsight before running formal evaluations. This lets you quickly spot routing issues or bad SQL generation before committing ground-truth pairs.

-- Call the agent via SQL
SELECT SNOWFLAKE.CORTEX.COMPLETE(
  'AGENT',
  ARRAY_CONSTRUCT(
    OBJECT_CONSTRUCT('role', 'user', 'content', 'What were total sales last month?')
  ),
  OBJECT_CONSTRUCT('agent_name', 'DEV_DB.CORTEX_AGENTS.EXAMPLE_AGENT')
) AS response;

Snowflake Intelligence shortcut

In Snowsight, navigate to AI & ML > Intelligence to chat with your agent through the built-in UI — no SQL needed. This is also the quickest way to demo the agent to a stakeholder before it's surfaced in Teams or a custom app.

Evaluate & Schedule

Steps 5 and 6 close the loop: evaluate accuracy against real questions before shipping to users, then automate the entire pipeline on a Snowflake Task schedule so models and agents stay current as data changes.

Step 5 — Run Evaluations

Accuracy gate before go-live

Target ≥95% answer_correctness before surfacing the agent to end users. The evaluation framework scores responses against your ground-truth pairs using Cortex LLM-as-judge — no external tooling required.

1 Build the ground-truth dataset — seeds/eval_ground_truth.csv
View eval_ground_truth.csv on GitHub

Add rows to seeds/eval_ground_truth.csv. Each row is a question the agent should be able to answer plus the correct response. The evaluation framework compares the agent's live output against these expected answers.

Column Purpose Example
input_query The natural language question What were total sales for January 2026?
ground_truth_json JSON with ground_truth_output key containing the expected answer {"ground_truth_output": "Total sales for January 2026 were $1.2M..."} 
# Load seeds and rebuild the eval dataset table
dbt seed
dbt run --select eval_dataset
2 Configure the evaluation — evaluations/example_eval_config.yml
View example_eval_config.yml on GitHub

The eval config YAML controls which agent to evaluate, which dataset to use, and which metrics to score. It gets uploaded to a Snowflake stage before running.

# dataset block: ONLY include on the first run.
# Remove it after the first successful run to avoid
# "dataset already exists" errors.
dataset:
  dataset_type: "CORTEX AGENT"
  table_name: "DEV_DB.CORTEX_AGENTS.EVAL_DATASET"
  dataset_name: "AGENT_EVAL_DATASET"
  column_mapping:
    query_text: "INPUT_QUERY"
    ground_truth: "GROUND_TRUTH"

evaluation:
  agent_params:
    agent_name: "EXAMPLE_AGENT"
    agent_type: "CORTEX AGENT"
  run_params:
    label: "v1 baseline"
    description: "Baseline evaluation before shipping to users"
  source_metadata:
    type: "dataset"
    dataset_name: "AGENT_EVAL_DATASET"

metrics:
  - "answer_correctness"     # Primary gate metric — target >= 95%
  - "logical_consistency"    # Secondary — checks internal coherence

Remove the dataset: block after the first run

The dataset only needs to be registered once. Leave the block in and subsequent runs will fail with a "dataset already exists" error. Delete or comment out the entire dataset: section after the first successful evaluation.

3 Run the evaluation — macros/run_evaluation.sql
View run_evaluation.sql on GitHub

The run_evaluation macro creates the staging infrastructure (file format + stage), uploads your eval config YAML, and calls EXECUTE_AI_EVALUATION('START', ...). Evaluation results appear in Snowsight under AI & ML > Evaluations.

# Run evaluation for the agent
dbt run-operation run_evaluation \
  --args '{agent_name: example_agent, run_name: v1, config_file: example_eval_config.yml}'
-- The macro issues this call internally:
CALL EXECUTE_AI_EVALUATION(
  'START',
  OBJECT_CONSTRUCT('run_name', 'v1'),
  '@DEV_DB.CORTEX_AGENTS.EVAL_CONFIG_STAGE/example_eval_config.yml'
);

Results are available in Snowsight under AI & ML > Evaluations. Review per-question scores and drill into failures before proceeding to scheduling.

Step 6 — Schedule with Snowflake Tasks

4 Create daily build and evaluation Tasks

Chain two Tasks: the first rebuilds all dbt models nightly, the second runs the evaluation immediately after. Both use EXECUTE DBT PROJECT to invoke the deployed dbt PROJECT object — no external orchestrator needed.

-- Task 1: nightly dbt build (prod target)
CREATE OR REPLACE TASK daily_cortex_build
  WAREHOUSE = ANALYTICS_WH
  SCHEDULE = 'USING CRON 0 6 * * * America/New_York'
AS
  EXECUTE DBT PROJECT DEV_DB.CORTEX_AGENTS.CORTEX_LIFECYCLE
    ARGS = 'build --target prod';

-- Task 2: evaluation runs immediately after the build
CREATE OR REPLACE TASK daily_agent_evaluation
  WAREHOUSE = ANALYTICS_WH
  AFTER daily_cortex_build
AS
  EXECUTE DBT PROJECT DEV_DB.CORTEX_AGENTS.CORTEX_LIFECYCLE
    ARGS = 'run-operation run_evaluation
            --args "{agent_name: example_agent,
                    run_name: daily,
                    config_file: example_eval_config.yml}"';

-- Activate both tasks
ALTER TASK daily_agent_evaluation RESUME;
ALTER TASK daily_cortex_build RESUME;

Task dependency order

Always RESUME child tasks before parent tasks. Snowflake requires dependent tasks to be active before the root task can be enabled. Resume daily_agent_evaluation first, then daily_cortex_build.

Shipping Surfaces

Once the agent is scheduled and consistently passing evaluations, it can be surfaced to end users through any of these channels — without changes to the agent itself.

Where Users Reach the Agent

Surface Best for How to connect
Snowflake Intelligence Snowsight-native chat UI — analysts and power users Create an Intelligence conversation, add your agent as a source
Microsoft Teams Business users who live in Teams — broadest reach Install the Snowflake Teams app, configure the agent endpoint
Custom app via MCP Embedded in a portal, Streamlit app, or internal tool Call the Cortex Agents REST API; MCP server available for local dev
Cortex Agent REST API Programmatic access from any language, backend service, or batch job POST to the agents/<name>:run endpoint with a PAT or OAuth token

Calling the Cortex Agent REST API

Every Cortex Agent object is automatically callable over REST — it is the lowest-level surface, and the same endpoint that Snowflake Intelligence, Teams, and MCP all sit on top of. Any language or service that can make an authenticated HTTPS request can drive the agent directly, which makes it the right choice for embedding the agent in backend services, scheduled jobs, or a custom front end.

curl -X POST \
  "https://<account>.snowflakecomputing.com/api/v2/databases/DEV_DB/schemas/CORTEX_AGENTS/agents/EXAMPLE_AGENT:run" \
  -H "Authorization: Bearer $SNOWFLAKE_PAT" \
  -H "Content-Type: application/json" \
  -d '{
    "messages": [
      { "role": "user",
        "content": [ { "type": "text", "text": "What were total sales last month?" } ] }
    ]
  }'

Authentication and streaming

Authenticate with a Programmatic Access Token (PAT) or OAuth bearer token scoped to a role that has USAGE on the agent. The endpoint streams its response as server-sent events (SSE), so your client can render the agent's reasoning, tool calls, and final answer incrementally rather than waiting for the full payload.

The agent object is the integration point

All of these surfaces call the same underlying Cortex Agent object — changes to instructions, tools, or the semantic view propagate everywhere automatically on the next alter_agent run. No surface-specific configuration is needed when you update the agent.
